# Research — Mitigación de riesgos operativos de gomemory

No quedaron marcadores `[NEEDS CLARIFICATION]` en el spec — el alcance ya se validó con el usuario durante la investigación previa (3 subagentes Explore + 1 subagente Plan sobre el estado real del código). Este documento consolida las decisiones técnicas concretas para cada mejora, en el formato Decision/Rationale/Alternatives.

## 1. Búsqueda: FTS5 con fallback a LIKE

- **Decision**: Agregar `CREATE VIRTUAL TABLE IF NOT EXISTS memory_search USING fts5(title, content, memory_id UNINDEXED)` en `migrate()` (`db.go:90-193`), alimentada por **dual-write manual** (INSERT/DELETE explícitos junto a las escrituras de `memories`), exactamente como `code_search` se alimenta hoy desde `code_graph.go:109,141-142` — no triggers SQL. Ranking vía `bm25()` en una función `searchMemoriesFTS`, con `searchMemoriesLike` (el código actual de `SearchMemories`, `memory.go:297-319`) como fallback si el build no tiene FTS5 compilado (mismo patrón que `code_graph.go:256-258`).
- **Rationale**: Es el único patrón de búsqueda de texto completo que el proyecto ya usa y ya sabe tolerar en builds sin FTS5 (pure-Go SQLite vía `modernc.org/sqlite` puede o no traer el tag de compilación). Reutilizar el patrón evita introducir una segunda forma de resolver el mismo problema.
- **Alternatives considered**: (a) Embeddings/búsqueda semántica vectorial — descartado: requeriría una dependencia nueva (proveedor de embeddings o librería ANN), desproporcionado para 0–135 filas por proyecto. (b) Índice invertido hecho a mano en Go — descartado: reinventa lo que FTS5 ya resuelve de forma nativa en SQLite. (c) Dejar el LIKE como está — descartado porque el usuario pidió explícitamente mitigar este riesgo, aunque de baja prioridad.

## 2. Backup automático

- **Decision**: Nueva función `CreateSnapshot(memRepo, relRepo, project, backupDir, keep)` en `application/usecases/backup.go`, que llama a `ExportProject` + `EncodeBundle` (ya existentes en `portability.go:18-63`, sin modificarlos), escribe el resultado en `<DataHome>/backups/<project-key>/<timestamp>.json`, y poda los archivos más antiguos si se supera `keep`. Se invoca desde `hookSessionEnd` (`cmd_hook.go:91`) en modo best-effort: cualquier error se descarta silenciosamente, igual que el resto de esa función (p. ej. `deps.SessionRepo.End` no propaga errores hacia el usuario).
- **Rationale**: Reutilizar el código de export existente evita duplicar lógica de serialización y mantiene el bundle en el mismo formato que ya sabe importarse con `mem import`. El directorio de backups vive bajo `DataHome()` (`globalstore.go:89-113`), ya resuelto y usado por el resto del proyecto — no se introduce una nueva noción de "dónde vive el estado de gomemory".
- **Alternatives considered**: (a) Copiar el archivo `mem.db` crudo — descartado: un archivo SQLite en modo WAL puede quedar en un estado no consistente si se copia mientras hay una transacción abierta; el bundle JSON vía `ExportProject` sí refleja un snapshot consistente al momento de la llamada. (b) Backup en cada escritura (por memoria guardada) — descartado: demasiado frecuente para el valor que aporta, y overhead innecesario en la ruta caliente de `save_memory`. (c) Backup vía cron/systemd/launchd — descartado: agrega una dependencia de configuración externa por plataforma; el hook de fin de sesión ya se ejecuta en el momento natural (fin de una unidad de trabajo) sin configuración adicional del usuario.

## 3. Redacción de secretos + hardening de permisos

- **Decision**: Función hermana `RedactSecrets(s string) string` en `domain/redact.go`, con una lista fija de regex (AWS `AKIA[0-9A-Z]{16}`, GitHub `gh[pousr]_[A-Za-z0-9]{36,}`, OpenAI/Anthropic `sk-(ant-)?[A-Za-z0-9-_]{20,}`, Slack `xox[baprs]-[A-Za-z0-9-]+`, JWT `eyJ[A-Za-z0-9_-]+\.[A-Za-z0-9_-]+\.[A-Za-z0-9_-]+`, bloques PEM `-----BEGIN[^-]*PRIVATE KEY-----.*?-----END[^-]*PRIVATE KEY-----`), llamada junto a `RedactPrivate` en los mismos call sites de escritura (`memory.go:36-37,267-268`, `session.go:39`). Permisos: `0700` en los directorios de data-home/proyecto (hoy `0755` en `globalstore.go:216`, `db.go:42,46`) y `0600` explícito sobre el archivo `mem.db` vía `os.Chmod` tras `sql.Open` (el driver `modernc.org/sqlite` no expone un modo de creación).
- **Rationale**: Mismo modelo mental que `RedactPrivate` — reemplazo de texto por regex, sin scoring de entropía ni clasificador, coherente con "Simplicidad Primero". El hardening de permisos es un cambio de una línea por sitio, sin lógica nueva, con beneficio de seguridad no trivial en máquinas multi-usuario.
- **Alternatives considered**: (a) Escáner de entropía genérico (detectar cualquier string "parece aleatorio") — descartado: alta tasa de falsos positivos sobre memorias legítimas (hashes, IDs, nombres técnicos), y contradice la Historia de Usuario 2 ("no debe degradar memorias legítimas"). (b) Cifrado en reposo (SQLCipher/age) — descartado explícitamente en el spec (Assumptions): defiende contra un modelo de amenaza (disco robado, root de otro usuario) que no aplica a una herramienta local de un solo autor; agrega gestión de claves y UX de desbloqueo desproporcionadas. (c) Redacción también en lectura — descartado por ahora: cambiaría la superficie de todas las herramientas de lectura (`get_memory`, `list_memories`, `search_memories`); se documenta como posible follow-up, no parte de esta feature.

## 4. Convención de compatibilidad documentada

- **Decision**: Comentario explícito en `migrate()` (`db.go`) fijando la disciplina de migraciones solo-aditivas ya seguida en la práctica (3 cambios de columna aditivos hasta hoy, cero destructivos). Comentario junto a `domain.ExportVersion` (`portability.go:5` — nota: el research previo ubicó esta constante en `domain/portability.go`, confirmar ruta exacta al implementar) fijando que cualquier cambio de campo en el bundle requiere subir la versión y una función de migración explícita (`migrateBundleVxToVy`).
- **Rationale**: No hay ninguna migración destructiva pendiente ni tabla `schema_version` — añadir bookkeeping ahora sería prematuro. La documentación en el propio punto del código sobrevive mejor que un documento externo porque un agente o el propio autor la ve exactamente donde va a romper la regla si no la sigue.
- **Alternatives considered**: (a) Tabla `schema_version` + framework de migraciones (`golang-migrate`) — descartado explícitamente en el spec: infraestructura para un problema (migraciones destructivas, múltiples consumidores) que el proyecto no tiene todavía. (b) Versionado de esquema de herramientas MCP — descartado: no hay evidencia de que múltiples clientes MCP dependan de una versión de tool schema específica hoy; es especulativo.
