El problema

El agente olvida todo al cerrar la sesión

Cada vez que abres una sesión nueva, el modelo arranca en blanco: no recuerda las decisiones de arquitectura, ni por qué se resolvió aquel bug, ni las convenciones del proyecto. gomemory cierra esa brecha.

Sin memoria

Repites contexto en cada sesión. Las decisiones viven solo en tu cabeza. El agente comete el mismo error dos veces.

→

Con gomemory

El contexto histórico se inyecta solo al arrancar. El agente guarda lo importante mientras trabaja y lo recupera cuando lo necesita.

¿En qué puerto se levanta el MCP? En ninguno. gomemory no abre un puerto TCP. El cliente MCP lanza mem mcp como subproceso y hablan por stdin/stdout con mensajes JSON-RPC (transporte stdio). Cuando la sesión termina, el proceso se cierra. Esto lo hace portable e idéntico en Linux, macOS y Windows, sin sockets ni configuración de red.
🧩

Arquitectura hexagonal

domain → reglas puras · application → casos de uso y puertos · adapters → MCP, CLI, SQLite.

🔒

Aislado por proyecto

Cada proyecto tiene su propia mem.db, pero no dentro del repo: viven todas en un store global, una subcarpeta por proyecto. La clave es el raíz de git (o el cwd). Nada se mezcla entre repos.

📦

Local y sin nube

Todo vive en tu máquina, en archivos SQLite. Sin cuentas, sin servidores externos, sin telemetría.

Aclaración importante — ¿dónde está realmente la base de datos? No está en el .memory/ de tu proyecto, sino en un store global del usuario: ~/.local/share/gomemory en Linux/macOS (o $XDG_DATA_HOME), %LOCALAPPDATA%\gomemory en Windows, o lo que fijes en GOMEMORY_DATA_HOME. La ruta exacta es <store>/projects/<clave-del-proyecto>/mem.db, donde la clave = slug de la carpeta + hash de la ruta absoluta (dos proyectos con el mismo nombre nunca colisionan).

¿Y el .memory/ que sí ves en el repo? Solo guarda archivos auxiliares: el marcador de sesión de los hooks y un context.md legible. La base de datos no se versiona ni ensucia tu árbol de trabajo. Si actualizas desde una versión antigua, un .memory/mem.db legado se migra solo al store global en el primer uso.
Los dos caminos hacia la memoria

El modelo decide · el sistema recuerda

gomemory graba memoria por dos vías complementarias. Una la activa el propio modelo; la otra ocurre sola, de forma determinista, sin gastar tokens del agente.

Camino A · Activo

Tools MCP

El modelo decide llamar a una herramienta cuando toma una decisión, corrige un bug o quiere buscar contexto previo. Son las 14 tools que el servidor expone.

  • Guardar: save_memory
  • Consultar: search_memories, get_context
  • Sesión: start_session / end_session
  • Rico pero voluntario: depende de que el modelo lo llame
Camino B · Pasivo

Hooks del agente

Eventos del ciclo de vida (arranque, fin de turno, plan aprobado…) disparan mem hook <evento> automáticamente. Es la red de seguridad determinista.

  • Inyecta el contexto histórico al arrancar
  • Registra actividad (archivos, comandos) sin pedírselo al modelo
  • Captura el plan aprobado aunque el modelo no lo guarde
  • Recuerda al modelo guardar si lleva rato sin hacerlo

🛡️ Regla de oro de los hooks

Un hook nunca puede abortar el arranque del agente. Ante cualquier error sale con código 0 y, como mucho, sin salida. La memoria es una ayuda, jamás un obstáculo.

Flujo de una sesión

Qué ocurre, paso a paso

Así se encadenan los hooks a lo largo de una sesión de trabajo. Cada evento de Claude Code invoca un subcomando del binario.

SessionStart→mem hook session-start

Arranque. Inicia la sesión si no hay ninguna activa e inyecta el contexto histórico (preferencias, sesiones previas, conflictos, memorias enlazadas) como additionalContext. El agente empieza ya sabiendo dónde quedó todo.

UserPromptSubmit→mem hook user-prompt-submit

Cada prompt. En el primer prompt fuerza la carga de las tools de memoria y recuerda el protocolo. En los siguientes es pasivo, salvo que toque un recordatorio de guardado (con debounce). Además guarda el prompt del turno como provenance.

PostToolUse · ExitPlanMode→mem hook plan-approved

Al aprobar un plan. Un turno de plan mode no toca archivos, así que el checkpoint normal lo descartaría por vacío. Este hook guarda el plan aprobado como memoria type=decision, sin gastar tokens.

Stop→mem hook turn-end

Fin de turno. Registra un checkpoint automático determinista: qué archivos se editaron y qué comandos corrieron, y reindexa los .go tocados. Los turnos de puro chat no generan checkpoint. Además refresca el snapshot del grafo externo si está viejo (v1.18.1) y, al pasar el umbral de huella, emite un aviso neutral de compactación (v1.20).

SubagentStop→mem hook subagent-stop

Fin de un subagente. Captura la actividad del subagente (que el hook Stop principal no ve, porque vive en su propio transcript). Misma red de seguridad, un nivel más abajo.

SessionStart · compact→mem hook post-compact

Tras compactar el contexto. Cuando la conversación se resume, este hook re-inyecta las instrucciones de recuperación y el contexto previo — su salida sobrevive a la compactación.

SessionEnd→mem hook session-end

Cierre. Cierra la sesión activa como red de seguridad. El resumen rico lo aporta el modelo llamando a end_session; aquí solo se garantiza que la sesión quede cerrada.

Blueprint de decisiones

Qué decide cada hook

La lógica vive en Go (transversal a todos los agentes); cada agente la invoca con su propia señal. Esta es la decisión concreta que toma cada uno.

Novedad v1.17 · memoria sináptica

Sinapsis: la memoria se enlaza sola

Guardar una memoria no la deja suelta. En el mismo punto por el que pasa todo guardado (MCP, checkpoint, CLI, TUI), gomemory teje una arista hacia la decisión más reciente de la sesión. Sin tokens del agente, de forma determinista.

El ancla es el engrama sustantivo más reciente de la misma sesión — nunca un checkpoint. Así se hila el reguero de decisiones y cada checkpoint queda colgado de la decisión que lo gobierna, sin ruido checkpoint↔checkpoint.

Sus garantías

  • ▸ Determinista — reglas fijas, no depende de que el modelo lo pida
  • ▸ Sin coste — cero tokens del agente
  • ▸ Idempotente — no duplica una arista ya existente
  • ▸ Best-effort — una sinapsis fallida jamás rompe el guardado
  • ▸ Transversal — vale para todas las vías de guardado

Al arrancar, estas conexiones vuelven al agente bajo 🔗 Sinapsis (memorias enlazadas) dentro del contexto histórico.

Novedad v1.20 · huella de contexto

Menos tokens, la misma señal

La memoria no sirve si inunda la ventana de contexto. La v1.20 recorta la huella del contexto inyectado sin perder valor: en el binario real, get_context pasó de ~86 KB a ~17 KB (~80% menos). Cuatro palancas, activadas por defecto.

1. Presupuesto

El contexto se acota a un límite de caracteres. Cada memoria se resume a su extracto con un puntero → get_memory <id>. El protocolo y los conflictos nunca se recortan.

2. Divulgación progresiva

search_memories y list_memories ya no vuelcan el contenido íntegro: devuelven extractos. El detalle completo se pide bajo demanda con get_memory.

3. Aviso de compactación

Un contador honesto de bytes emitidos por sesión dispara, al pasar un umbral, un recordatorio neutral (sin comandos de un agente concreto, con debounce). Agnóstico al agente.

4. Dedup en la fuente

Al guardar, una topic_key (o la identidad tipo+título) detecta duplicados dentro de una ventana y evita grabar la misma decisión una y otra vez.

Los tunables (presupuesto, umbral, ventana de dedup) viven en settings.json con valores por defecto sensatos: la feature va activada sin configurar nada, y un valor negativo es opt-out explícito. Generar el contexto nunca altera las memorias.
Novedad v1.19 · portabilidad y control

Llévate la memoria · gestiónala a mano

La memoria de un proyecto ya no está atada a una sola máquina, y su control deja de vivir solo en la línea de comandos.

📤

Export / import portable

mem export vuelca las memorias y sus relaciones a un JSON cross-OS; mem import las reinyecta en otro equipo. Deduplica por hash (tipo+título+contenido), preserva timestamps y no crea sinapsis espurias.

⌨️

Pantalla de Configuración (TUI)

En la TUI, la tecla c abre Configuración: estado del grafo externo y su toggle, auto-approve y accesos a export/import.

En una frase

El resumen para presentar

gomemory es un binario en Go que actúa como servidor MCP por stdio (sin puerto) y como ejecutor de hooks. Guarda la memoria de cada proyecto en un SQLite local aislado. El modelo graba y consulta memoria llamando a sus 14 tools; en paralelo, 7 hooks inyectan el contexto histórico al arrancar y capturan la actividad de forma determinista — de modo que, aunque el modelo olvide guardar, la memoria del proyecto no se pierde. Cada memoria se enlaza sola (sinapsis), el contexto puede enriquecerse con un grafo de código externo enchufable, y su huella se mantiene bajo control; todo es portable con export/import.
